�� MEDIUM | Logic Correctness | utils/NostrConnectUtils.ts:546-566, 589-592
   
   Title: Fallback Payment Hash Generation - Potential Collision Risk
   
   Location: extractPaymentHashFromActivity + buildDeterministicPaymentHash
   
   Description:
   When primary payment hash sources unavailable (paymentHash field, decoded invoice),
   falls back to deterministic hash: sha256(`zeus-nwc-fallback:${timestamp}-${amount}`)
   
   Risk Scenario:
   - Two payments created at same Unix second with identical amount
   - No payment hash in either payment (missing from LN backend)
   - No invoice available for decoding
   - Both generate same fallback hash
   - Activity dedup treats as single transaction
   - User sees one transaction instead of two
   
   Probability: LOW (99% of payments have hash or decodable invoice)
   
   Fallback Chain:
   1. payment.paymentHash (usually populated)
   2. extracted from invoice (bolt11 decode)
   3. extracted from activity.invoice.payment_hash
   4. buildDeterministicPaymentHash(key) ← collision risk here
   5. activity.id or ''
   
   Recommendation:
   - Option A: Include millisecond precision: `zeus-nwc-fallback:${Date.now()}-${amount}`
   - Option B: Include payment.id: `zeus-nwc-fallback:${payment.id || uuid()}-${amount}`
   - Option C: Add logging to detect fallback usage in production
   
   Acceptable As-Is: YES
   Rationale: Fallback is defense mechanism, not primary path. Activity field exists 
             to disambiguate cases where hash collides. 99% case fully covered.

🟡 MEDIUM | Logic Design | models/NWCConnection.ts:290-310

   Title: Budget Renewal - Absolute Interval vs Calendar-Aligned Behavior
   
   Location: needsBudgetReset computed getter, BUDGET_RENEWAL_MS constant
   
   Description:
   Budget renewal uses absolute time intervals:
   - 'daily': exactly 86,400,000 ms (24 hours in UTC)
   - User in timezone UTC+9 creating at 23:00 local might expect reset at 23:00 next day
   - Actually resets 24 hours from creation timestamp (in UTC time)
   - Net effect: May reset at 15:00 UTC+9 next day depending on creation time
   
   Current Implementation:
   ```
   get needsBudgetReset(): boolean {
     const elapsedMs = Date.now() - this.lastBudgetReset.getTime();
     const intervalMs = BUDGET_RENEWAL_MS[this.budgetRenewal!];
     return elapsedMs >= intervalMs;
   }
   ```
   
   Design Analysis:
   ✓ CORRECT: Simple, predictable, no DST edge cases
   ✓ CORRECT: Matches "daily" semantics (24-hour period)
   ✗ NOT: Calendar-aligned (e.g., midnight local time)
   
   Risk: LOW - Behavioral difference, not data corruption
         User may be surprised by exact reset time
   
   Recommendation:
   - Document in UI: "Budget resets every 24 hours from creation time"
   - If calendar-alignment desired (business requirement), requires timezone handling
   - Current absolute-interval design is safer
   
   Acceptable As-Is: YES
   Rationale: Simpler than calendar-aligned, no time-of-day edge cases
